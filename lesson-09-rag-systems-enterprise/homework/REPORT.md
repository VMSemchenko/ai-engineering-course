# Відкриття «Breaking Point» у RAG системах

## 1. Що вимірювалось і гіпотеза
Був проведений бенчмаркінг RAG-системи, масштабуючи її корпус від 1K до 300K документів (passages) з датасету MS MARCO. 
Вимірювалися наступні показники:
- **Recall@1 та Recall@10** для оцінки деградації пошуку при зростанні шуму.
- **Latency (p95)** для оцінки деградації продуктивності лінійного пошуку (Flat Index).
- **RAM та Disk** для оцінки ресурсів.

**Робоча гіпотеза**: При масштабуванні recall@1 деградує швидше за recall@10 через "змивання" релевантних документів семантично схожими дистракторами. Latency зростає лінійно (`O(N)`) і стає bottleneck'ом.

## 2. Де і чому система зламалася? (Точка перелому)
Точка перелому (Breaking Point) чітко проявилася на переході від **100K до 300K** документів:
- **Latency (Flat Index)** виросла лінійно і досягла **~10ms (p95)** на 300K. Хоча це ще прийнятно, але екстраполяція на 1M+ дасть >30ms, що вже є bottleneck'ом для production.
- **Recall@1 (Dense)** просів катастрофічно: від **0.98 (на 1K)** до **0.89 (на 100K)** і врешті-решт до **0.76 (на 300K)**. 
- **Recall@10** деградував значно повільніше (релевантний документ залишається у топ-10, хоча втрачає 1-шу позицію).

Гіпотеза щодо Recall@1 та Latency повністю підтверджена!

## 3. Який Fix реалізовано і наскільки він допоміг
Було реалізовано два механізми Fix:
1. **Для Latency**: Заміна `IndexFlatIP` на **FAISS HNSW** (`IndexHNSWFlat`). 
   - *Результат*: Latency впала з 9.92ms до **0.46ms** на 300K. Час пошуку став майже логарифмічним/константним. Це повністю вирішує проблему швидкості.
2. **Для Recall**: Використання **Hybrid Search (BM25 + Dense Embeddings + RRF)**.
   - *Результат*: **Гіпотеза щодо Hybrid Search не підтвердилася у цій базовій конфігурації!** Hybrid показав Recall@1 = 0.54 (проти 0.76 у Dense). Це пояснюється тим, що `bge-small-en-v1.5` є дуже потужною dense-моделлю для MS MARCO, тоді як базова токенізація (`split()`) для BM25 додала занадто багато лексичного "шуму", який через RRF знизив позицію релевантних документів.

## 4. Рекомендації для Production на 1M+ корпусі
1. **Векторний Індекс**: На 1M+ використання HNSW є обов'язковим. Flat Index буде займати >30ms.
2. **Retrieval**: Для покращення Recall не можна покладатися лише на базовий BM25. Необхідно використовувати `Cross-Encoder Reranker` (наприклад `BAAI/bge-reranker-v2-m3`) на top-50/100 результатах HNSW, оскільки Reranker ефективно відсіює дистрактори.
3. **Гібридний пошук**: Якщо застосовувати Sparse + Dense, необхідно використовувати потужний токенізатор (stemming, stop-words) для BM25 або використовувати нейромережевий Sparse (як SPLADE), оскільки наївний BM25 лише зашкодить якості (як показав тест).
